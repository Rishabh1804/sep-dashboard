# Excellent Data Capture — The North Star

**Phase:** 7
**Status:** LOCKED — 7 May 2026
**Authority:** Every Phase 7 design decision is measured against this bar.

---

## Why This File Exists

> *"This whole thing relies on excellent data capture, that's the unit on which everything will work."*

"Excellent" is a vibe unless we make it a measurable bar. This file names the seven dimensions of excellence and the metrics that test them. Every Phase 7 form, UI shell decision, and infrastructure choice is justified relative to one or more of these dimensions.

---

## The Seven Dimensions

### 1. Speed

**Definition:** Time from intent ("I need to log this") to commit ("server has the record").

**Measurable bar:**
- Common entry (Production Entry with valid pre-fill) ≤ 3 effective taps
- Worker check-in/out ≤ 2 effective taps via top-bar chip
- Form open → submit ≤ 5 seconds for routine entries
- Submit confirmation visible/audible/haptic within 200ms of tap

**Implementation alignment:**
- Smart defaults from prior entry (pre-fill defense bundle)
- Big-button picker grids (no typing for high-frequency fields)
- 64px touch targets (no mis-tap retry)
- Numpad always on screen for numeric fields (no keyboard switch lag)
- Submit-and-continue default for repeated forms

### 2. Accuracy

**Definition:** Probability that submitted data reflects reality (no typos, no wrong-job mis-attribution, no fat-finger range errors).

**Measurable bar:**
- Wrong-job mis-attribution rate ≤ 1% (vs. predicted 8-12% without defenses)
- DFT >50 µm rejected at Zod boundary (impossible value)
- Quantity outside 2σ of rolling mean prompts confirmation
- Mis-tapped picker rows ≤ 5% (vs. typeahead's 10-20% on gloved hands)

**Implementation alignment:**
- Zod runtime validation at storage Layer 2 (per Phase 6)
- Sanity hard-blocks on numeric fields
- Visual diff highlight on pre-filled fields
- Server-side `authoredBySelf` rule (cannot fake author)
- Multi-modal submit confirmation (visual + audio + haptic)

### 3. Completeness

**Definition:** Coverage of operational data that future role-apps will eventually own. Until specialized apps ship, the handler captures everything.

**Measurable bar:**
- All 9 forms shipping in alpha
- Adoption rate ≥95% across all forms by Week 4 of rollout
- Every operational write (job, production, DFT, dispatch, stock, machine state, worker shift, note) has a form
- Forms cover 100% of paper-notebook scope

**Implementation alignment:**
- 9 forms scoped per [HANDLER_FORMS.md](HANDLER_FORMS.md)
- Adoption rate metric (Week 4 KPI)
- Parallel paper run during rollout (Weeks 0-4) reconciles divergence

### 4. Reliability

**Definition:** Probability that a submitted entry actually lands in the source-of-truth Firestore.

**Measurable bar:**
- Online write commit success rate ≥ 99.9%
- Offline writes survive PWA close/reopen (queue persists)
- 7-day idle PWA reopens with queue intact (Android-only constraint per Phase 7)
- Form draft auto-saved every 200ms during input
- No silent data loss across PWA reinstall (pre-flush confirmation per Phase 5)

**Implementation alignment:**
- Firestore offline-first persistence (built-in)
- Form draft auto-save to IndexedDB
- Pre-flush confirmation on app open
- Idempotency keys + ±5min server dedup window
- Android-only handler PWA (Phase 7 BLOCKER fix)
- App Check (Phase 4) prevents unauthenticated/throttled writes from succeeding silently

### 5. Forensics

**Definition:** Ability to reconstruct what happened, who did what, when, from which build.

**Measurable bar:**
- Every write tagged with: `author_user_id`, `app_version`, `created_at` (server-set), `idempotency_key`
- Every state change generates an audit_event (post-write CF)
- Every revision (edit-with-reason) preserves prior state in `revisions[]` array
- Every CF action logs structured info to Cloud Logging
- Topic digest hourly cron commits to git → time-traveled state for any past hour

**Implementation alignment:**
- Phase 4 rules pin `author_user_id` + `created_at` immutable on update
- Phase 5 audit_events generated by CF post-write trigger on every collection
- Phase 5 `revisions[]` append-only (never destructive resolution)
- Topic digest hourly cron (Phase 2)
- Daily Firestore export to GCS (Phase 6)
- Per-write app_version stamping

### 6. Recoverability

**Definition:** Ability to fix mistakes without data loss; ability to recover from system failures.

**Measurable bar:**
- Bad migration recoverable from `_migrations_backup/` within 30 days
- Bad CF logic recoverable via replay/rebuild admin tool
- Stolen-phone offline writes rejected via `active_token_id` rotation; legitimate work surfaceable in rejected-write inbox
- Steward edit-with-reason for handler entries beyond 24h window
- Pending-conflicts inbox surfaces lost edits for review

**Implementation alignment:**
- TTL'd `_migrations_backup/` collection (Phase 6 must-add)
- Daily Firestore export to GCS (Phase 6 must-add)
- CF replay/rebuild admin tool (Phase 5 must-add)
- `active_token_id` rotation + rejected-write inbox (Phase 8 must-add)
- Edit-with-reason workflow (Phase 8)
- Pending-conflicts inbox (Phase 5)

### 7. Adoption-friendly

**Definition:** Workers actually use the app; they don't revert to paper notebooks.

**Measurable bar:**
- Worker can pick up the app with <1 hour of guided onboarding
- Doesn't fight the notebook mental model
- Adoption rate ≥95% by Week 4 of rollout
- Zero conflict modals on handler PWA (per Phase 5; conflicts route to dashboard inbox)
- i18n: Devanagari labels primary; English secondary toggle; long-press TTS

**Implementation alignment:**
- Phased rollout with parallel paper run (Phase 7 / [ADOPTION_PLAN.md](ADOPTION_PLAN.md))
- Devanagari + iconography + audio TTS (Phase 7 BLOCKER fix)
- Form-with-smart-defaults primitive for check-in/out (≤2 effective taps)
- Multi-modal submit confirmation (visual + audio + haptic)
- Daily 10-minute huddle for first 2 weeks
- Handler-as-trainer KPI = adoption rate (NOT entries-recorded)

## How This Bar Operates

When designing a new form, refactoring a flow, or evaluating a proposal:

1. **Name the dimensions it touches.** Most decisions touch multiple.
2. **Show the measurable bar for each.** Don't argue from vibes.
3. **Profile against the bar pre-launch.** UI mock studies, prototype testing, dry-run.
4. **Track post-launch.** Metrics dashboards surface where the bar is missed.
5. **Tune the bar over time.** As real data accumulates, calibrate target values.

---

## Anti-Patterns (avoid)

These all *sound* good but violate the bar:

- **"Confirmation modal for every submit"** → kills speed
- **"Free-text reason for every edit"** → kills accuracy (becomes "fixed")
- **"Email/password auth for handlers"** → kills speed + adoption-friendly
- **"Fully featured offline-first with conflict detection on phone"** → kills adoption-friendly (workers don't have time/context for conflict modals on factory floor)
- **"Hire designer to make it pretty"** → tactical, not strategic; doesn't move any bar
- **"AI-suggest based on context"** → could violate accuracy (AI gets things wrong, worker submits unchanged)
- **"Add gamification"** → can violate accuracy (gaming the metric); careful design needed
- **"Voice notes as primary form input"** → trade-off: improves speed, but transcription burden falls on steward; rejected for alpha per Phase 7
